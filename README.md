# ParserService

## Обновления и функционал сервиса

В текущей версии сервис превратился в полноценную систему обработки данных с поддержкой асинхронных очередей (NATS) и автоматизированным парсингом через Playwright.

### Основные компоненты

-   **Движок парсинга (`ParserCore`):** Использует `Playwright` для эмуляции браузера и получения актуальных данных. Логика парсинга изолирована по операторам (например, `Dertour`).
    
-   **Система очередей (NATS):** Сервис использует NATS для получения команд на запуск парсинга и передачи результатов в другие части системы.
    
-   **Обработка ошибок:** Выделенный воркер (`ErrorLoggingWorker`) перехватывает ошибки парсинга и сохраняет их в `ErrorLogConnection` для дальнейшего анализа.
    
-   **API (Endpoints):** Для управления и мониторинга используется OpenAPI/Scalar.
    
### API Эндпоинты

| Метод | Эндпоинт | Описание |
| :--- | :--- | :--- |
| **GET** | `/` | Редирект на документацию API (Scalar). |
| **GET** | `/operators` | Список доступных туроператоров. |
| **POST** | `/parser/run` | Запуск парсера для оператора. |
| **POST** | `/references/update` | Обновление справочников (стран, регионов, валют). |

### Что за что отвечает (архитектура)

-   **`ParserRunnerHandler`**: Основная точка входа для запуска задач. Принимает запрос, вызывает нужный парсер и отдает данные в репозиторий.
    
-   **`TourDataRepository`**: Слой доступа к данным. Отвечает за логику обновления существующих цен (`existingPrice`) или создания новых (`newPrice`).
    
-   **`PageProcessor`**: Низкоуровневый компонент для работы с HTML-страницами через Playwright.
    
-   **`OperatorInitializationService`**: При старте приложения подтягивает необходимые настройки для каждого туроператора.
    

### Технические примечания

1.  **Часовые пояса:** Все даты (`Date`, `UpdatedAt`) в базе данных хранятся в формате `timestamptz` (UTC). При записи данных сервис принудительно устанавливает `DateTimeKind.Utc`.
    
2.  **Миграции:** Схема БД управляется через `EF Core Migrations`. При добавлении новых полей используй `dotnet ef migrations add <Name>`.
    
3.  **Окружение:** Настройки подключения (`PostgresConnection`, `NatsConnection`) подтягиваются из переменных окружения.

    ├── Entities/   # Сущности (модели таблиц)
    └── Migrations/ # Сюда EF Core автоматически складывает миграции

```

### Иерархия локаций и справочников

База данных спроектирована с жесткой изоляцией данных в разрезе каждого туроператора и использует каскадную структуру от общего к конкретному:


```
[operators]
   │
   ├──► [currencies] (Валюты оператора)
   │
   ├──► [placements] (Варианты размещения: Adults/Children)
   │
   └──► [countries] (Страны оператора)
           │
           └──► [tours] (Туры/Категории внутри страны)
                   │
                   └──► [regions] (Регионы в туре)
                           │
                           └──► [hotels] (Отели в регионе)

```

## Структура хранения цен (оптимизация под Highload)

Хранение цен разделено на две таблицы: **price_types** (уникальные комбинации маршрутов и отелей) и **prices** (динамические сетки цен). Такая денормализация позволяет избежать дублирования гео-данных для миллионов записей и обеспечивает высокую скорость фильтрации.


```
[price_types] (Справочник уникальных маршрутов)
       │
       └───► [prices] (Миллионы динамических цен)
                ├──► [placements] (Размещение)
                └──► [currencies] (Валюта)

### 1. Таблица price_types (Комбинации параметров)
Эта таблица выступает в роли «хэша» или уникального ключа для комбинации мест отправления, прибытия и конкретного отеля. Если у туроператора есть 1000 разных цен на разные даты в один и тот же отель из одного и того же города, в этой таблице будет всего одна запись.

* Все связи настроены с поведением DeleteBehavior.Restrict, чтобы защитить географические справочники от случайного каскадного удаления.

| Имя поля в БД       | Тип данных (EF Core / Postgres) | Описание                                     | Поведение связи (FK) |
| :------------------ | :------------------------------ | :------------------------------------------- | :------------------- |
| Id                  | int (serial)                    | Первичный ключ                               | Principal Key        |
| DepartureCountryId  | int                             | ID страны вылета                             | Restrict             |
| DepartureRegionId   | int                             | ID региона/города вылета                     | Restrict             |
| ArrivalCountryId    | int                             | ID страны прибытия                           | Restrict             |
| ArrivalRegionId     | int                             | ID региона прибытия                          | Restrict             |
| HotelId             | int                             | ID конкретного отеля                         | Restrict             |

### 2. Таблица prices (Сетка цен)
Самая высоконагруженная таблица в системе, куда парсер непрерывно записывает актуальные прайсы. При удалении комбинации параметров (price_types) все связанные с ней цены удаляются автоматически (DeleteBehavior.Cascade).

| Имя поля в БД | Тип данных (EF Core / Postgres)       | Описание                                     | Поведение связи (FK) |
| :------------ | :------------------------------------ | :------------------------------------------- | :------------------- |
| Id            | long (bigserial)                      | Первичный ключ (64-bit для больших объемов)  | Principal Key        |
| PriceTypeId   | int                                   | Ссылка на уникальную комбинацию параметров   | Cascade              |
| PlacementId   | int                                   | Ссылка на тип размещения (2AD, 2AD+1CH...)   | Restrict             |
| CurrencyId    | int                                   | Ссылка на валюту оператора                   | Restrict             |
| Price         | decimal(18,2) (numeric)               | Актуальная стоимость тура                    | —                    |
| Date          | DateOnly (date)                       | Дата вылета/начала тура                      | —                    |
| Nights        | int (integer)                         | Количество ночей в туре                      | —                    |
| UpdatedAt     | DateTime (timestamp)                  | Локальное время обновления (для дедупликации)| —                    |

## Производительность и Индексы (Индексация таблицы prices)

Для обеспечения мгновенной выборки на фронтенде и быстрой работы парсера при обновлении данных, на таблице prices настроены два ключевых индекса:

1. Составной индекс IX_prices_PriceTypeId_Date_Nights
   * Поля: [PriceTypeId, Date, Nights]
   * Зачем нужен: Позволяет парсеру за константное время проверить, есть ли уже в базе цена на эту конкретную дату и количество ночей для данного отеля. Если есть — мы её обновляем, если нет — добавляем новую. Это исключает дублирование строк.

2. Одиночный индекс IX_prices_UpdatedAt
   * Поля: [UpdatedAt]
   * Зачем нужен: Необходим для фоновых воркеров очистки. Позволяет одним быстрым запросом найти и удалить «протухшие» цены, которые парсер не обновлял (например, старше 24 часов), поддерживая базу в актуальном и неперегруженном состоянии.
